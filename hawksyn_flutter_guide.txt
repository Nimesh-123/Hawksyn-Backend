### HAWKSYN FLUTTER IMPLEMENTATION SPECIFICATIONS (FOR AI / ANTIGRAVITY) ###

PURPOSE: Implement the core analysis flow including Cases, Intents, Test Payments, and CV Attachment.

---

### 1. CONFIGURATION & MODELS
AI Task: Create these Dart models to match backend JSON structures.

- Base URL: http://192.168.1.20:5000/api
- Auth: Always include header { "Authorization": "Bearer <TOKEN>" }

[CaseModel]
- caseId (String)
- caseName (String)
- caseDescription (String?)
- minPrice (double)
- defaultCurrency (String)

[IntentModel]
- intentId (String)
- intentName (String)
- intentHorizonDays (int)
- isAvailable (bool) -> Maps from 'isActive' in backend config
- availabilityLabel (String) -> "Available" or "Coming Soon"
- isDefault (bool)

[PlaybookModel]
- playbookVersionId (String)
- cvMandatory (bool)
- allowedCvFormats (String)

---

### 2. CORE LOGIC & UI STATES (SCREEN 1 & 2)

#### SCREEN A: CASE SELECTION (Home)
- Trigger: GET /api/cases
- Display: List of CaseModel.
- UI: Show caseName and "Starts from <Price> <Currency>".
- Action: On tap, navigate to Intent Screen with selected caseId.

#### SCREEN B: INTENT SELECTION
- Trigger: GET /api/cases/{caseId}/intents
- UI LOGIC (CRITICAL):
  IF intent.isAvailable == false:
    - Display UI with opacity 0.5.
    - Show "Coming Soon" badge/text.
    - On Tap: Show Alert Dialog: "This feature is coming soon! We're building it for you."
  IF intent.isAvailable == true:
    - Display normal UI (Opacity 1.0).
    - If intent.isDefault == true, show a "Recommended" star or pill.
    - On Tap: Proceed to PAYMENT FLOW.

---

### 3. PAYMENT FLOW (THE RUN CREATION)

#### STEP 1: LOAD PLAYBOOK (Validation)
- Call: GET /api/cases/{caseId}/intents/{intentId}/playbook
- Note: Store 'cvMandatory' value to know if user needs to upload PDF later.

#### STEP 2: INITIATE TEST PAYMENT
- Call: POST /api/payment/initiate
- Body: { "caseId": "...", "intentId": "...", "platform": "test" }
- Store: 'purchaseId' from response.

#### STEP 3: VERIFY & CREATE RUN
- Call: POST /api/payment/verify
- Body: { "purchaseId": "...", "caseId": "...", "intentId": "..." }
- IMPORTANT: Response returns 'runId'. Save this locally.

---

### 4. CV ATTACHMENT FLOW (AFTER PAYMENT)

Screen appears ONLY IF cvMandatory was true.

#### OPTION A: KEEP EXISTING CV
- Button: "Continue with Last Uploaded CV"
- Call: POST /api/runs/{runId}/cv/keep-existing (Empty Body)
- Success: Move to analysis progress screen.

#### OPTION B: UPLOAD NEW CV
- Button: "Upload New PDF" (Allow max size 10MB)
- Call: POST /api/runs/{runId}/cv/upload
- Body: Multipart request with key 'cv' containing the PDF file.
- Success: Move to analysis progress screen.

---

### 5. ERROR HANDLING SPEC
- 401: Navigate to Login screen.
- 403: If payment verify fails, notify user to contact support.
- 404: Show "Service Temporarily Unavailable".
- 500: Generic "Something went wrong" with a retry button.

### 6. PROFILE REVIEW & CONFIRMATION FLOW

This screen appears after the CV is parsed and attached to the Run.

#### STEP 1: FETCH CLEAN PROFILE (GET /profile)
- Call: `GET /api/v1/runs/{runId}/profile`
- UI Display:
  - **personalInfo**: Name, email, phone, location, role title.
  - **experience**: List of work history.
  - **projects**: List of key projects and tech stacks.
  - **skills**: Technical, soft, and languages.
  - **education & certifications**: Academic and professional background.
  - **seniority & employment**: AI-inferred professional standing.
  - **assumptions**: **CRITICAL** - Show a list of items for user review.
    - Each item has `field`, `assumedValue`, and `reason`.
    - UI should allow user to "Confirm" or "Edit" these.

#### STEP 2: SAVE EDITS & CONFIRM (PUT /profile)
- Call: `PUT /api/v1/runs/{runId}/profile`
- Body:
```json
{
  "profile": {
    "identity": { "fullName": "..." },
    "composition": { "skills": { "languagesSpoken": ["English"] } }
  },
  "assumptionsReview": [
    { "aeuId": "AEU_INF_001", "action": "CONFIRMED" },
    { "aeuId": "AEU_INF_003", "action": "CORRECTED" }
  ]
}
```
- **LOGIC**:
  - Send **only** changed fields in the `profile` object (Destructive empty arrays are now blocked by backend safety checks, but partial updates are cleaner).
  - Backend will automatically check `mandatoryFieldsStatus`.

- **Success (200 OK)**:
  - If response has `isConfirmed: true` and `isLocked: true`:
    - Show "Profile confirmed successfully."
    - Navigate to the Analysis Progress screen.
    - The screen is now locked for editing.

- **Error (400 Bad Request)**:
  - Occurs if mandatory fields are missing.
  - Response will include `missingFields: ["current_role", "skills", ...]`.
  - UI: Highlight missing fields and show an error toast: "Please fill all mandatory fields before confirming."

#### STEP 3: LIKING THE STATE
- Once `isLocked: true` is received, the frontend must block any further `PUT` requests for this `runId`.
